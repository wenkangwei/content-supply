# Content Supply Web - v1.0 技术难点与优化路径

> 版本: v1.0 | 日期: 2026-05-08

---

## 一、已识别难点与落地方案

### 难点 1: 大量内容列表渲染性能

**问题**: 内容池可能有数千条内容，一次性加载会卡顿

**v1.0 方案**: 传统分页（`skip` + `limit`），每页 20 条

**优化路径**:
- v1.1: 虚拟滚动（`@tanstack/react-virtual`），支持万级内容流畅滚动
- v1.1: 无限滚动（Intersection Observer），替代翻页
- v2.0: 本地 IndexedDB 缓存已加载内容，支持离线浏览

### 难点 2: 暗色主题精准还原 AIHOT 风格

**问题**: AIHOT 使用深蓝黑渐变、半透明卡片、精细的层级关系

**v1.0 方案**: TailwindCSS 自定义颜色 + CSS 变量，5级背景层级

**优化路径**:
- v1.1: 毛玻璃效果（`backdrop-blur`）增强层次感
- v2.0: 亮色主题支持，CSS 变量切换

### 难点 3: 异步任务状态更新

**问题**: 爬取/改写是异步操作，需要展示进度

**v1.0 方案**: 定时轮询（5s），任务完成后自动停止

**优化路径**:
- v1.1: WebSocket 实时推送，后端需要配合升级
- v2.0: SSE (Server-Sent Events)，比 WebSocket 轻量

### 难点 4: 内容搜索体验

**问题**: 后端搜索是 POST 请求 + 全文模糊匹配，无法做到即时搜索

**v1.0 方案**: 防抖搜索（300ms），搜索按钮触发

**优化路径**:
- v1.1: 本地缓存已加载内容的标题/摘要，支持即时前端筛选
- v2.0: 接入全文搜索引擎（如 MeiliSearch）

### 难点 5: 清理审核工作流交互

**问题**: 清理操作涉及批量删除，需要明确的确认流程

**v1.0 方案**: Modal 确认 + 审核人必填 + 状态过滤

**优化路径**:
- v1.1: 批量审核（多选 + 一键确认/拒绝）
- v2.0: 审核历史时间线、操作回滚

## 二、v1.0 功能边界（不做什么）

| 不做 | 原因 |
|------|------|
| 用户认证/登录 | 后端无认证，本地部署 |
| 虚拟滚动 | v1.0 分页足够，v1.1 加入 |
| WebSocket | 后端未实现，v1.1 加入 |
| PWA/离线 | 非核心，v2.0 加入 |
| 国际化 | 初期中文为主，v1.1 加入 |
| 数据导出 | 非核心，v2.0 加入 |
| 明亮主题 | 默认暗色，v2.0 加入 |

---

## 三、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 后端API不兼容 | 低 | 高 | 严格按API文档开发，类型校验 |
| 性能问题（大量内容） | 中 | 中 | 分页策略，React.memo |
| AIHOT风格还原不精确 | 中 | 低 | 提取核心设计元素，不强求像素级 |
| 移动端体验差 | 低 | 低 | 响应式断点，移动端基础适配 |

---

## 四、Feature 拆分原则

按照 agent-harness-framework 范式:

1. 每个 Feature 独立可测试
2. 一个 Feature = 一个 Agent session
3. 优先级: setup → 核心展示 → 管理功能 → 优化打磨
4. 每个 Feature 必须端到端验证后标记 `passes: true`
5. 修复优先于新功能开发
